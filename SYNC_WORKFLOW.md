# Руководство по инкрементальному бэкапу

## Быстрый старт

### Первая настройка (рабочий ПК)

1. Скачайте скрипты из Dashboard → Data Sync Procedures
2. Поместите скрипты в корневую папку проекта
3. Установите пароль (опционально, но рекомендуется):
   ```cmd
   set BACKUP_PASSWORD=ваш_надежный_пароль
   ```

### Ежедневный workflow

#### На рабочем ПК (в конце дня)

```cmd
cd your-project
backup_work_delta.bat
```

- Первый запуск: Создаёт полный бэкап (~MB-GB в зависимости от проекта)
- Последующие запуски: Создаёт дельту (обычно KB-MB)
- Результат: `sys_log_chunk.bin`

#### Передача

1. Откройте веб-интерфейс LocalGitMirror
2. Перейдите в File Manager → Shared Folders
3. Загрузите `sys_log_chunk.bin`

#### На домашнем ПК

1. Скачайте `sys_log_chunk.bin` из File Manager
2. Поместите в папку проекта
3. Запустите:
   ```cmd
   cd your-project
   restore_home_delta.bat
   ```

## Как это работает

### Технология инкрементальной дельты

- **Отслеживание состояния**: Файл `.last_sync` хранит хеш последнего коммита
- **Первый запуск**: Полный бандл со всей историей
- **Инкрементальный**: Только коммиты с последней синхронизации
- **Сжатие**: Максимальное сжатие 7-Zip + AES-256
- **Результат**: Минимальные размеры файлов (обычно 1-5% от полного репозитория)

### Пример размеров файлов

```
Полный репозиторий: 500 MB
Первый бэкап: ~450 MB (сжатый)
Ежедневная дельта: ~2-10 MB (только изменения)
```

### Функции безопасности

- Шифрование AES-256
- Шифрование заголовка (скрывает структуру файлов)
- Нейтральное имя файла (`sys_log_chunk.bin`)
- Защита паролем

## Устранение неполадок

### "Failed to create incremental bundle"

**Причина**: Нет новых коммитов с последнего бэкапа

**Решение**: Сделайте коммиты сначала, или удалите `.last_sync` для полного бэкапа

### "Failed to apply bundle"

**Причина**: Локальные несохранённые изменения

**Решение**: 
```cmd
git status
git stash
restore_home_delta.bat
git stash pop
```

### "Invalid checkpoint hash"

**Причина**: Повреждённый файл `.last_sync`

**Решение**: Удалите `.last_sync` и запустите полный бэкап

## Лучшие практики

1. **Коммитьте перед бэкапом**: Всегда коммитьте свою работу сначала
2. **Храните .last_sync**: Не удаляйте этот файл (отслеживает состояние)
3. **Устанавливайте пароль env**: Избегайте ввода пароля каждый раз
4. **Регулярные бэкапы**: Ежедневно или после значительной работы
5. **Проверяйте восстановление**: Проверьте `git log` после восстановления

## Расширенное использование

### Пользовательский пароль для каждого бэкапа

```cmd
backup_work_delta.bat MySecurePassword123
```

### Проверка того, что будет забэкаплено

```cmd
git log --oneline $(cat .last_sync)..HEAD
```

### Ручная проверка бандла

```cmd
git bundle verify data.bundle
```

## Интеграция с существующим workflow

Эта система работает вместе с вашим обычным Git-workflow:

- **Рабочий ПК**: Обычные коммиты → Ежедневный дельта-бэкап
- **Передача**: Зашифрованный чанк через веб-интерфейс
- **Домашний ПК**: Восстановление дельты → Продолжение работы
- **Повтор**: Двусторонняя синхронизация по мере необходимости

Не конфликтует с:
- Git-репозиториями (GitHub, GitLab и т.д.)
- Ветвистыми workflow
- Существующими решениями для бэкапа

## Справочник файлов

- `backup_work_delta.bat` - Создаёт инкрементальные бэкапы
- `restore_home_delta.bat` - Применяет бэкапы
- `.last_sync` - Отслеживание состояния (автогенерируемый)
- `sys_log_chunk.bin` - Зашифрованный результат (передавайте этот)
- `data.bundle` - Временный (автоудаляемый)
